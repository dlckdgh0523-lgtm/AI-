# 쇼핑 컨시어지 — 법령 GraphRAG 기반 AI 커머스 에이전트

> 상황 기반 상품 추천·비교 + 소비자권리(환불·청약철회·광고규제) 안내를 하나의 멀티 도구 에이전트로 제공하는 웹서비스 프로토타입.

**데모 URL**: (배포 후 기입)

---

## 1. 문제 정의

온라인 쇼핑에서 소비자가 겪는 문제는 두 층위입니다.

1. **구매 전** — 검색 필터로는 "장마철 원룸에서 쓸", "자취 시작하는데" 같은 *상황*을 표현할 수 없고, 상품·후기가 너무 많아 비교가 어렵다.
2. **구매 후** — "개봉했는데 환불 되나?", "반품비는 누가 내나?" 같은 질문의 답은 법령에 있지만, 소비자는 전자상거래법 제17조를 읽지 않는다. 챗봇이 법령 근거 없이 답하면 환각 위험이 있고, 원칙 조문만 보고 답하면 **예외 조항을 놓쳐 틀린 답**을 준다.

이 프로토타입은 두 문제를 하나의 에이전트로 풉니다: 자연어 입력 → 에이전트가 스스로 도구(상품 검색·후기 수집·법령 그래프 검색)를 선택·조합 → 근거를 인용한 답변. 모든 판단 과정은 **트레이스 UI**로 투명하게 공개됩니다.

## 2. 왜 GraphRAG인가 — 측정으로 답함

법령은 태생적으로 그래프입니다. `법령 → 조 → 항 → 호` 계층과 "제17조에 따라", "~를 준용한다" 같은 참조 관계로 연결되어 있습니다. 단순 벡터 RAG는 질문과 유사한 조문 하나는 잘 찾지만, **완전한 답에 필요한 원칙+예외+절차 조문 묶음**은 놓칩니다.

이 프로젝트는 벡터 검색으로 진입점을 찾고, `REFERS_TO` 참조 엣지를 1홉 순회해 관련 조문을 함께 회수합니다. 효과는 자체 평가 하네스([eval-results.md](./eval-results.md), `npm run eval`)로 측정했습니다:

| 지표 (벡터 진입 예산 동일, topK=4) | 벡터 단독 | GraphRAG |
|---|---|---|
| 단일 조문 검색 recall (12문항) | 11/12 (92%) | 11/12 (92%) |
| **복합 질의 완전성 (5문항)** | 2/5 (40%) | **5/5 (100%)** |

단일 조문 질문에서는 차이가 없습니다 — 정직하게 기록합니다. GraphRAG의 가치는 복합 질의에서 나타나며, 5문항 중 3건은 그래프 확장이 없었으면 정답 조문 일부를 놓쳤습니다.

**대표 사례**: "개봉한 노트북 단순변심 환불 돼?" → 벡터 검색은 제17조 제1항(원칙: 7일 내 철회 가능)만 잡지만, 그래프 확장은 제2항 각호(예외: 가치 훼손 시 제한)와 그 단서("확인을 위한 개봉은 제외"), 그리고 "판매자가 철회 제한을 표시하지 않았으면 여전히 철회 가능"이라는 예외의 예외까지 함께 회수해, 에이전트가 3단 논리로 답합니다.

## 3. 아키텍처

```
사용자 자연어 입력
   │
   ▼
Next.js /api/chat ── 오케스트레이터 (Claude, adaptive thinking, SSE 스트리밍)
   │                  · 질문 분해 → 전문 에이전트 위임 → 보고 합성
   │                  · 복수 위임은 같은 턴에 병렬 실행 (Promise.all)
   │            ┌─────────────┴─────────────┐
   │            ▼ (병렬)                     ▼ (병렬)
   │     🛒 쇼핑 전문 에이전트          ⚖️ 법률 전문 에이전트
   │     · 독립 tool use 루프           · 독립 tool use 루프
   │     · search_products             · search_law (GraphRAG)
   │     · search_reviews                       │
   │     (네이버 쇼핑/블로그 API)          Neo4j AuraDB
   │                                   · 법령 4종, 조문 186개, 참조 엣지 258개
   │                                   · 벡터 인덱스(Voyage 임베딩) + 풀텍스트
   │                                   · 벡터 진입 → REFERS_TO 1홉 확장
   ▼
트레이스 UI (오케스트레이터 판단·위임 과제·에이전트별 도구 호출을 레인으로 실시간 표시)
   +
Supabase (대화 로그) ──▶ /admin 커머스 인텔리전스 대시보드
```

**대시보드 = 에이전트 데이터로 비즈니스 인사이트 생성** (`/admin`). 단순 로그 뷰어가 아니라, 이 프로젝트만의 자산인 법령 그래프를 비즈니스 인사이트로 연결한다:
- **소비자 이슈 핵심 법령 조문**: 법령 지식그래프에서 참조(REFERS_TO) 중심성이 높은 조문 = 소비자 분쟁이 가장 많이 얽히는 핵심 조항 (예: "부당한 표시·광고 금지" 9회 참조). 순수 Cypher, 대화 로그 없이도 표시.
- **고객 니즈 분석**: 실제 대화 질문을 카테고리(환불/추천/비교/광고/약관)로 분류해 어떤 니즈가 많은지.
- **신뢰성 지표**: 법령 인용 검증 통과율, 의미 캐시 히트, PII 마스킹 적용.
- 도구 호출 분포, 토큰/비용 관측.

> 화면의 모든 데이터는 실데이터다 — Neo4j 그래프, 네이버 API, Supabase 로그, 실시간 에이전트. mock/placeholder 없음.

**에이전트 설계 (agent-as-tool 패턴)**: 오케스트레이터에게 서브 에이전트는 `delegate_shopping` / `delegate_law`라는 도구입니다. 각 전문 에이전트는 역할별 시스템 프롬프트와 도구 부분집합으로 독립 루프를 돌고 보고서만 반환합니다. 역할별 컨텍스트 분리로 프롬프트가 짧고 정확해지며, 두 영역이 섞인 질문("이어폰 추천해주고 환불 규정도")에서 병렬 위임으로 레이턴시를 줄입니다. 비용 제어로 서브 에이전트는 `effort: medium` + 도구 호출 상한을 적용했습니다 (복합 질문 기준 회당 약 $0.79 → $0.30).

### 그래프 스키마 (온톨로지)

```cypher
(:Law {name, mst, enforcedAt})
  -[:HAS_ARTICLE]-> (:Article {key, articleNo, title, content, embedding})
  -[:HAS_PARAGRAPH]-> (:Paragraph {no, content})
(:Article)-[:REFERS_TO]->(:Article)   // 조문 본문에서 "제N조" 참조를 파싱해 생성
```

데이터는 **법제처 국가법령정보센터 Open API**에서 현행 법령 XML을 받아 구축합니다 (전자상거래법·표시광고법·약관규제법·소비자기본법). 항상 최신 현행 법령 기준입니다.

### 기술 선택 근거

| 선택 | 이유 |
|---|---|
| Neo4j (그래프+벡터 통합) | 별도 벡터 DB 없이 "벡터 진입 → 그래프 확장"이 DB 하나에서 완결. GDS 없는 Free 티어 제약은 순수 Cypher로 해결 |
| Claude tool use + adaptive thinking | 에이전트의 도구 선택 판단을 모델에 위임하고, 판단 요약을 트레이스로 노출 |
| 프롬프트 캐싱 | 시스템 프롬프트+도구 정의를 캐싱해 매 루프 입력 비용 ~90% 절감 (대시보드에서 cache_read 확인 가능) |
| SSE 스트리밍 | 텍스트·사고과정·도구 호출을 이벤트 단위로 실시간 전달 |

## 5. 신뢰성·보안 설계 (적대적 검토)

에이전트가 법령을 인용하고 신뢰 불가한 외부 텍스트(후기)를 다루므로, "그럴듯하지만 틀린 답"과 "오염된 입력"을 각각 방어합니다.

### 적대적 인용 검증 ([verify.ts](./src/lib/verify.ts))

법률 에이전트의 보고서는 사용자에게 가기 전 2단계 검증을 거칩니다.
1. **결정론적 대조** — 인용된 "○○법 제N조"가 이번 조사에서 실제 그래프에서 검색된 조문 집합에 존재하는지 코드로 확인.
2. **LLM 크리틱** — "이 인용을 반박하라"는 임무의 검증관이 조문 원문과 주장을 대조.

실패 시 1회 자동 수정하고, 검증 자체가 예외 나면 원본으로 폴백해 단일 장애점이 되지 않습니다. 실전에서 검색되지 않은 조문(제19조), 그래프에 없는 타법 조문(민법 제580조)을 인용하는 환각을 자동 검출·수정하는 것을 확인했습니다.

### 프롬프트 인젝션 레드팀 ([redteam.ts](./scripts/redteam.ts), [결과](./redteam-results.md))

후기/상품 데이터에 공격 4종(위장 canary, 유령 상품 삽입, 개인정보 유도, 경쟁사 비방)을 주입하고 방어 OFF/ON을 비교했습니다. **정직하게 기록한 결과**:

| 모델 | 방어 OFF | 방어 ON |
|---|---|---|
| Opus 4.8 (프로덕션) | 0/4 | 0/4 |
| Haiku 4.5 (약한 모델) | 1/4 | 0/4 |

- 프로덕션 모델은 방어 없이도 견고해 방어의 효과가 여기서 측정되지 않았습니다 — 이를 숨기지 않고 negative result로 기록합니다.
- 방어의 실효성은 약한 모델에서 드러나며(1건→0건), 방어층은 다층 방어(defense-in-depth)로 유지합니다.
- **측정 방법론 자체도 한 번 틀렸다가 고쳤습니다**: 초기 문자열 매칭 판정은 방어된 에이전트가 조작 시도를 *경고*하려고 공격 문자열을 언급하는 것을 "공격 성공"으로 오판했습니다. "지시를 실제로 이행했는가"를 판정하는 LLM 저지로 교체했습니다.

### 토큰 비용 최적화 ([cost-optimization.md](./cost-optimization.md), `npm run bench`)

멀티 에이전트를 유지하면서 비용을 줄이는 레버를 적용·측정했습니다.
- **서브에이전트 모델 다운시프트 (주요 레버)**: 오케스트레이터는 Opus 4.8 유지, 조사 담당 서브에이전트는 Haiku 4.5로 전환 → 복합 질문 1건 $0.45 → $0.09 (**80% 절감**), 품질 유지 확인.
- **프롬프트 캐싱**: 시스템 프롬프트+도구 정의 캐싱으로 반복 루프 입력 절감.
- **의미 캐시**: Neo4j 벡터 인덱스로 반복 법령 질문의 보고서를 재사용. **캐시 키를 사용자 원 질문으로 잡는 것이 관건** — 처음엔 LLM이 생성한 가변 검색 질의를 키로 써서 히트율이 낮았으나, 안정적인 사용자 질문을 키로 바꾸자 의미가 같은 질문("환불 며칠?" ≈ "청약철회 기간?")이 실제로 히트합니다(유사도 0.906~0.918 히트, 다른 쟁점 ≤0.784 정상 미스). 히트 시 법률 서브에이전트 루프 전체를 스킵합니다. "인스타 백엔드 팁(Redis)을 그대로 베끼지 않고 캐싱 원리만 그래프 DB로 구현"한 사례입니다.

### 채택하지 않은 것 — Task Budgets

Anthropic의 최신 beta 기능(에이전트 루프 토큰 예산)을 검토했으나 도입하지 않았습니다. 하드 캡이 아니라 모델이 자율 조절하는 *권고*라 실질 통제는 기존 `MAX_ITERATIONS` + `max_tokens`가 이미 담당하고, 최소 예산(20K 토큰)이 서브 에이전트 규모에 과합니다. "최신이라서 넣는" 대신 기존 통제로 충분하다고 판단했습니다.

### 데이터 소스의 한계를 알고 설계 (상품 평판)

상품 카드는 네이버 쇼핑 검색 API의 실데이터(제목·가격·이미지·판매처)로 렌더링하고, **가격 범위 필터는 실제로 동작**합니다. 그러나 **네이버 검색 API는 별점·리뷰 개수·구매 횟수를 제공하지 않습니다.** 따라서 "리뷰 100건 이상" 같은 필터는 만들지 않았고(가짜 수치를 지어내지 않기 위해), 제품 평판은 블로그 후기(search_reviews)의 **정성적 근거**로만 제시합니다. 서브에이전트 프롬프트에 "별점·리뷰 수를 지어내지 말 것"을 명시했습니다. 별점·리뷰 수가 꼭 필요하면 쿠팡/11번가 파트너스 API(별도 승인)나 크롤링(차단 리스크)이 필요한데, 프로토타입 범위에서는 "가격은 실필터, 평판은 후기 근거"가 정직하고 실용적인 선택입니다.

### 프롬프트의 알려진 한계 (적대적 자기검토)

- **최종 합성 답변은 인용 검증을 거치지 않습니다.** 인용 검증은 법률 서브에이전트 보고서에 적용되며, 오케스트레이터가 합성 중 조문을 바꾸면 그 부분은 검증 밖입니다. 프롬프트로 "전문가 보고의 사실을 새로 만들거나 바꾸지 말 것"을 강하게 지시해 완화했으나(프롬프트 레벨), 완전한 보장은 최종 답변에도 검증을 적용하는 것입니다 — 다음 단계 과제.
- **재검색 요청은 재위임하도록 프롬프트에 명시**했습니다("더 저렴한 걸로"를 직접 답하지 말고 delegate_shopping 재호출). 프롬프트 준수에 의존하는 부분이라 회귀 테스트로 지켜봐야 합니다.

## 6. 한계와 개선 방향

- **참조 파싱의 한계**: "제N조" 정규식 기반이라 타법 참조("「민법」 제○조"), 준용 관계의 방향성은 아직 구분하지 않음. 관계 타입 세분화(EXCEPTION_OF, APPLIES_MUTATIS_MUTANDIS)가 다음 단계.
- **평가셋 규모**: 17문항 · 저자 작성 gold. 소비자분쟁조정 사례 기반 확장과 LLM 저지 도입 여지.
- **후기 데이터**: 블로그 검색 스니펫 수준. 후기 본문 수집→임베딩하면 리뷰 RAG로 확장 가능.
