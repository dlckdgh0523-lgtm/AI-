# A/B: 단일 에이전트 vs 멀티 에이전트 오케스트레이션

- 평가일: 2026-07-06
- 질문: 상품+법령 복합 질문 3건 (오케스트레이션이 유리할 것으로 가정한 지점)
- 품질: LLM 저지 루브릭 (완전성/인용정확성/실행성, 각 1~5)

## 요약 (평균)

| 지표 | 단일 에이전트 | 멀티 에이전트 |
|---|---|---|
| 완전성 | 5.0 | 5.0 |
| 인용 정확성 | 5.0 | 5.0 |
| 실행성 | 5.0 | 5.0 |
| 입력 토큰(합) | 117739 | 237179 |
| 출력 토큰(합) | 7286 | 29858 |
| 평균 레이턴시 | 48.0s | 127.0s |

## 케이스별 품질 (완전성/인용/실행)

| 질문 | 단일 | 멀티 |
|---|---|---|
| 10만원 이하 무선 이어폰 추천해주고, 온라인에서 사면… | 5/5/5 | 5/5/5 |
| 자취 시작하는데 20만원 이하 로봇청소기 추천하고, 하… | 5/5/5 | 5/5/5 |
| 캠핑 버너 2개 비교해주고, 개봉 후 단순변심 반품 가… | 5/5/5 | 5/5/5 |

## 해석 (정직한 결론 — 예상과 달랐음)

**이 3문항에서 멀티 에이전트는 품질 우위를 보이지 못했다.** 단일·멀티 모두 5/5/5 만점으로 동률이고, 멀티는 입력 토큰 2배·레이턴시 2.6배(48s→127s)를 더 썼다. 즉 이 난이도의 복합 질문에서는 **잘 프롬프트된 단일 에이전트가 더 싸고 빠르면서 동등하다.**

몇 가지를 정직하게 짚는다.
- **저지 천장 효과**: 루브릭이 3축 1~5점이라 상한(5)에 몰리면 두 구조를 변별하지 못한다. "둘 다 훌륭함"까지만 말할 수 있고 "누가 더 나은가"는 이 측정으로 못 가린다 — 측정의 한계.
- **레이턴시 127초는 실서비스 UX 문제**다. 멀티 구조의 순차 위임+검증이 누적된 결과로, 이 자체가 개선 대상이다.
- **멀티가 정당화되는 조건은 따로 있다**: (a) 진짜 독립적인 작업의 병렬화 이득, (b) 역할별 모델 티어링(서브에이전트 Haiku)으로 **실 비용(원)**은 토큰 수와 달리 단일-전량-상위모델 대비 낮을 수 있음, (c) 도구·도메인이 늘 때의 컨텍스트 격리·확장성. 이 3문항은 그중 어느 것도 강하게 요구하지 않았다.

**결론**: "멀티 에이전트를 썼으니 더 낫다"는 주장은 이 데이터로 성립하지 않는다. 아키텍처 선택은 품질이 아니라 **확장성·비용 구조·병렬성**으로 정당화해야 하며, 단순 질의에는 단일 에이전트 경로를 두는 것이 옳다(향후 과제: 질문 복잡도로 라우팅).
