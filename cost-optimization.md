# 토큰 비용 최적화 (측정 기반)

멀티 에이전트 오케스트레이션을 유지하면서 비용을 줄이는 레버를 적용하고, 각각을 수치로 검증했다. 재현: `npm run bench`.

## 1. 서브에이전트 모델 다운시프트 (주요 레버, 검증됨)

오케스트레이터(질문 분해·최종 합성)는 판단력이 중요해 Opus 4.8을 유지하고, 서브에이전트(상품/법령 조사)는 도구 호출·요약 위주라 **Haiku 4.5로 다운시프트**했다.

| 항목 | Opus 서브에이전트 | Haiku 서브에이전트 |
|---|---|---|
| 복합 질문 1건 비용 | $0.45 | **$0.09** |
| 절감 | — | **80%** |

- 단가: Opus $5/$25, Haiku $1/$5 (per 1M in/out). 조사 작업은 큰 도구 결과를 입력으로 읽어 입력 토큰 비중이 커서, 입력 단가 1/5인 Haiku 전환 효과가 크다.
- 품질 확인: Haiku 서브에이전트로 전환 후에도 법령 인용 검증·비교표·"확인 불가" 정직 표기가 정상 동작함을 E2E로 확인.

## 2. 프롬프트 캐싱 (적용됨)

오케스트레이터·서브에이전트의 시스템 프롬프트+도구 정의에 `cache_control`을 적용. 루프 2회차부터 `cache_read`로 재사용(실측 로그에서 cache_read 토큰 확인). 반복 루프의 입력 비용을 절감한다.

## 3. 의미 캐시 (구현·검증했으나 한계 명확 — 정직한 기록)

법령 질문은 의미가 반복되므로("환불 며칠?" ≈ "청약철회 기간?"), 질문 임베딩이 유사하면 이전 GraphRAG 결과를 재사용하도록 Neo4j 벡터 인덱스로 의미 캐시를 구현했다 (별도 Redis 불필요 — 이미 있는 그래프 DB 재사용).

**임계값을 측정으로 결정** (`scripts/test-cache.ts`, `measure-sim.ts`):

| 질문 유형 | 임베딩 유사도 |
|---|---|
| 동일 문자열 | 1.000 |
| 같은 질문/패러프레이즈 | 0.897 ~ 0.918 |
| 다른 쟁점 (반품비 vs 환불기간) | 0.784 |
| 완전 다른 쟁점 (광고 규제) | 0.654 |

같은 질문군과 다른 질문 사이 마진이 ~0.11이라, 임계값 0.88로 "같은 질문은 히트 / 다른 질문은 false hit 차단"을 잡았다.

**한계 (숨기지 않고 기록)**: 실전에서 서브에이전트(LLM)가 같은 의도라도 매번 다른 질의를 생성해 유사도가 0.85~0.92로 흔들린다. 이는 다른 질문 밴드(≤0.78)와의 안전 마진에 가까워, 임계값을 더 낮춰 히트율을 올리면 **잘못된 법령 답을 캐시로 주는 위험**이 커진다. 따라서 의미 캐시의 실전 히트율은 태생적으로 낮고, 안전(정확성)을 우선해 보수적 임계값을 유지했다.

**개선 방향**: 캐시 키를 LLM이 생성한 가변 질의가 아니라 **사용자의 원 질문**(안정적)으로 잡으면 히트율이 오른다. 이는 다음 단계 과제로 기록한다.

## 종합

- **검증된 실질 절감은 모델 다운시프트(80%)**가 주도한다.
- 프롬프트 캐싱은 반복 루프 입력을 절감한다.
- 의미 캐시는 기법으로서 검증(정확한 임계값·false hit 차단)했으나, LLM 질의 가변성 때문에 실전 히트율이 낮다는 한계를 정직하게 남긴다. "최신 기법을 넣었다"보다 "적용하고 한계까지 측정했다"를 기록한다.
