# 토큰 비용 최적화 (측정 기반)

멀티 에이전트 오케스트레이션을 유지하면서 비용을 줄이는 레버를 적용하고, 각각을 수치로 검증했다. 재현: `npm run bench`.

## 1. 서브에이전트 모델 다운시프트 (주요 레버, 검증됨)

오케스트레이터(질문 분해·최종 합성)는 판단력이 중요해 Opus 4.8을 유지하고, 서브에이전트(상품/법령 조사)는 도구 호출·요약 위주라 **Haiku 4.5로 다운시프트**했다.

| 항목 | Opus 서브에이전트 | Haiku 서브에이전트 |
|---|---|---|
| 복합 질문 1건 비용 | $0.45 | **$0.09** |
| 절감 | — | **80%** |

- 단가: Opus $5/$25, Haiku $1/$5 (per 1M in/out). 조사 작업은 큰 도구 결과를 입력으로 읽어 입력 토큰 비중이 커서, 입력 단가 1/5인 Haiku 전환 효과가 크다.
- 품질 확인: Haiku 서브에이전트로 전환 후에도 법령 인용 검증·비교표·"확인 불가" 정직 표기가 정상 동작함을 E2E로 확인.

## 2. 프롬프트 캐싱 (적용됨)

오케스트레이터·서브에이전트의 시스템 프롬프트+도구 정의에 `cache_control`을 적용. 루프 2회차부터 `cache_read`로 재사용(실측 로그에서 cache_read 토큰 확인). 반복 루프의 입력 비용을 절감한다.

## 3. 의미 캐시 (구현·검증 완료, 실제 히트)

법령 질문은 의미가 반복되므로("환불 며칠?" ≈ "청약철회 기간?"), 유사한 질문이 오면 이전 법률 보고서를 재사용하도록 Neo4j 벡터 인덱스로 의미 캐시를 구현했다 (별도 Redis 불필요 — 이미 있는 그래프 DB 재사용). 법령은 요청마다 바뀌지 않으므로 법률 보고서 캐시는 안전하다(상품 추천은 시세 변동 때문에 캐시하지 않는다).

### 핵심 설계 결정: 캐시 키 = 사용자 원 질문

처음엔 서브에이전트(LLM)가 생성한 검색 질의를 키로 썼는데, LLM이 같은 의도라도 매번 다른 키워드 나열을 생성해(유사도 0.85~0.92로 흔들림) 히트율이 낮았다. **캐시 키를 LLM 가변 질의가 아니라 안정적인 사용자 원 질문으로 바꾸자 문제가 해결됐다** — 위임 레벨(`agents.ts`)에서 사용자 질문을 임베딩해 캐시한다.

**임계값·히트 검증** (`scripts/test-cache.ts`, `measure-sim.ts`):

| 질문 유형 | 임베딩 유사도 | 결과 (임계값 0.88) |
|---|---|---|
| 동일 문자열 | 1.000 | 히트 |
| 같은 질문/패러프레이즈 (사용자 표현) | 0.906, 0.918 | ✅ **히트** |
| 다른 쟁점 (반품비 vs 환불기간) | 0.784 | 미스 (정상) |
| 완전 다른 쟁점 (광고 규제) | 0.654 | ✅ **정상 미스** (false hit 없음) |

같은 질문군(0.90+)과 다른 질문(≤0.78) 사이 ~0.12 마진에 임계값 0.88을 두어, **의미가 같은 질문은 히트시키고 다른 질문에는 잘못된 법령 답을 주지 않는다**. 캐시 히트 시 법률 서브에이전트 루프 전체(Haiku 조사 + 그래프 검색 + 인용 검증)를 건너뛴다.

## 종합

- **모델 다운시프트(80%)** + **의미 캐시(반복 법령 질문 시 서브에이전트 루프 전체 스킵)** + **프롬프트 캐싱(반복 루프 입력 절감)**의 3단 비용 최적화.
- 의미 캐시의 관건은 "무엇을 키로 잡는가"였다 — LLM 출력이 아니라 안정적인 사용자 입력을 키로 잡아야 실전에서 히트한다.
